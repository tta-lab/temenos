Temenos Background Jobs

Design document — 2026-05-25

Problem

Temenos's POST /run and MCP bash tool are fully synchronous. The caller blocks until the sandboxed command finishes or the context deadline kills it. The default timeout is 120 seconds — long enough to waste agent turns on builds, installs, and test runs, but not long enough for genuinely slow work.

A short auto-background timeout (15s) lets fast commands return synchronously while long-running ones detach into pollable jobs. The agent gets immediate feedback and can choose to check back or continue other work.

Reference: Crush

Crush (the basis for Lenos) implements this pattern in-process:

  • Every command starts as a BackgroundShell via BackgroundShellManager
  • A 100ms polling loop waits up to DefaultAutoBackgroundAfter seconds
  • If done in time → return synchronously (job removed from manager)
  • If still running → return job ID + message: "Use job_output tool to view output or job_kill to terminate"
  • No partial output in the initial background response — model calls job_output if it wants to see stdout/stderr
  • Max 100 concurrent background jobs

Temenos adapts this pattern for a daemon architecture: the job registry lives in the daemon, exposed over the admin socket. Lenos polls from the outside.

Architecture

Socket Model

Two unix sockets, same daemon process:

SocketPathPermissionsEndpointsConsumer
Admin ~/.temenos/daemon.sock 0o600 /run, /health, /session/*, /jobs/* Lenos (privileged)
Job API ~/.temenos/job.sock 0o600 /jobs/* only Agents via temenos job CLI

The job socket is a strict subset of the admin socket — same handlers, fewer routes. Agents cannot reach /run or /session/*, so they cannot bypass sandbox restrictions by crafting arbitrary AllowedPaths.

DELETE /jobs/{id}

Kills a running background job. Returns 200 with the job's final state. Returns 404 if not found. Returns 409 if already completed.

Internal Implementation

Job Registry (internal/daemon/jobs.go)

// BackgroundJobManager manages background command executions.
type BackgroundJobManager struct {
    mu   sync.RWMutex
    jobs map[string]*BackgroundJob
}

type BackgroundJob struct {
    ID          string
    CallerID    string
    Command     string
    Status      JobStatus // "running" | "completed" | "killed"
    Stdout      *syncBuffer
    Stderr      *syncBuffer
    ExitCode    int
    StartedAt   time.Time
    CompletedAt time.Time
    cancel      context.CancelFunc
    done        chan struct{}
}

type syncBuffer struct {
    mu  sync.Mutex
    buf bytes.Buffer
}

Lifecycle

  1. Start — POST /run with auto_background_after > 0 calls jobManager.Start(ctx, callerID, command, execCfg, sbx). Returns immediately with a *BackgroundJob.
  2. Poll — Handler polls job.IsDone() every 100ms (like Crush) for up to auto_background_after seconds.
  3. Fast return — If done within threshold: remove from manager, return synchronous RunResponse.
  4. Background — If still running: return {job_id, status: "background"}. Job stays in manager.
  5. Completion — Background goroutine finishes, updates Status, ExitCode, CompletedAt.
  6. GC — Completed jobs retained for 5 minutes, then cleaned by Cleanup() (called on each new Start).

Limits

  • Max 100 concurrent background jobs per daemon
  • Output truncated at 64KB per stream (consistent with existing RunResponse)
  • Completed jobs GC'd after 5 minutes

CLI Commands

temenos job list

$ temenos job list
ID       COMMAND            STATUS     STARTED
a1b2c3   go build ./...     running    10:30:00
d4e5f6   npm install        completed  10:28:15

Defaults to status=running. Use --status all to show all jobs. Flags: --status, --caller-id.

temenos job log <id>

$ temenos job log a1b2c3
[stdout]
# github.com/tta-lab/temenos
cmd/temenos/main.go:12: undefined: foo

[stderr]
(exit code: 2)

Streams current output. If the job is still running, shows what's accumulated so far.

temenos job kill <id>

$ temenos job kill a1b2c3
killed job a1b2c3 (go build ./...)

What This PR Does NOT Cover

  • Lenos integration — polling goroutine, result injection, system prompt updates. Separate PR.
  • MCP background support — no consumers yet. Can be added later using the same job registry.
  • Persistent job state — jobs are in-memory only. Daemon restart kills running background jobs. Acceptable for current scale.
